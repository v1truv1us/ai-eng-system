# Repository Cleanup And Workflow Verification Plan

## Goal
- Do a safe cleanup of local stale artifacts.
- Keep `docs/` as the canonical documentation tree.
- Verify and tighten the repo's spec, ADR, and orchestration workflow story.

## Current State Summary
- Spec support already exists and is strong.
  - `content/commands/specify.md`
  - `content/commands/spec.md`
  - `skills/spec-driven-development/SKILL.md`
- Workflow support exists, but the surfaced path is inconsistent.
  - Canonical lifecycle commands exist for `research`, `specify`, `plan`, `work`, and `review`.
  - `verify` exists as `content/commands/verify.md`, but docs still list it as planned.
  - Docs claim alias coverage for `/plan` and `/review`, but only `spec` and `build` are implemented as standalone alias command files.
- ADR support exists, but the canonical path is split.
  - `content/commands/decision-journal.md` uses `docs/decisions/`
  - `skills/documentation-and-adrs/SKILL.md` uses `docs/adr/`
  - `build.ts`, `src/learning-automation/heuristics.ts`, and `tests/learning-automation.test.ts` already reinforce `docs/decisions/`
- Local cleanup opportunities are real and conservative.
  - Root-level historical scripts appear stale.
  - `INSTALLATION.md` and `PLUGIN.md` appear stale relative to active `docs/` content.
  - `docs/` and `docs-site/src/content/docs/` drift, but `docs-site/` should not be removed in the safe pass.

## Recommended Implementation

### 1. Declare Canonical Sources In Docs
Update contributor-facing docs so the repo clearly states:
- `docs/` is the canonical documentation source.
- `content/commands/` is the canonical command-definition source.
- `skills/` is the canonical skill-definition source.
- `.claude/`, `.opencode/`, `dist/`, `plugins/`, and `docs-site/src/content/docs/` are generated, mirrored, or published surfaces unless explicitly documented otherwise.

Primary files:
- `docs/index.md`
- `docs/reference/commands.md`
- `docs/spec-driven-workflow.md`
- `README.md`

### 2. Remove Only Strong Stale Script Candidates
Delete or archive only the files with strong evidence of being historical and unused:
- `validate-phase3.js`
- `validate-phase3-clean.js`
- `test-parameter-passing.js`
- `import-codeflow-agents.sh`
- `import-codeflow-agents-fixed.sh`
- `verify-install.sh`

Rule:
- If any active references remain, archive instead of delete.

### 3. Retire Stale Root Docs In Favor Of `docs/`
Archive or remove root docs that duplicate older installation/plugin guidance:
- `INSTALLATION.md`
- `PLUGIN.md`

Replace inbound links with current canonical docs under `docs/`.

### 4. Keep `docs-site/` Intact In This Pass
Do not remove `docs-site/src/content/docs/` during the safe cleanup.
Only document it as a non-canonical mirror/published surface so future cleanup can happen intentionally.

### 5. Archive Historical Narrative Docs, Keep Reference Docs Active
Move implementation-complete or fix-summary docs out of the active documentation surface when they are not part of current user guidance.

Likely archive candidates:
- `docs/ai-eng-ralph-cli-complete.md`
- `docs/ralph-wiggum-fix-complete.md`
- `docs/ralph-wiggum-execution-fix.md`
- `docs/ralph-wiggum-autonomous-looping-fix.md`

Keep active:
- `docs/spec-driven-workflow.md`
- `docs/reference/commands.md`
- other enduring reference and usage docs

### 6. Normalize ADR Support Around `docs/decisions/`
Use the smallest consistency fix: keep `docs/decisions/` as canonical because it is already aligned with commands, build, heuristics, and tests.

Update these sources to match:
- `skills/documentation-and-adrs/SKILL.md`
- `templates/decisions/decision.md` if needed
- `skills/graph-rag/SKILL.md` if example paths still mention `docs/adr/`
- `skills/multimodal-corpus-ingestion/SKILL.md` if example paths still mention `docs/adr/`
- any docs that still describe `docs/adr/`

Do not introduce or preserve both `docs/adr/` and `docs/decisions/` as active parallel conventions.

### 7. Tighten The Workflow Story To `spec -> plan -> work -> verify -> review`
Make the smallest set of fixes needed for a clean orchestration path.

Update:
- `content/commands/verify.md`
- `docs/reference/commands.md`
- `docs/spec-driven-workflow.md`
- `docs/reference/workflow-surface-matrix.md`
- `docs/reference/skills-first-map.md`

Specific goals:
- Treat `verify` as shipped, not planned.
- Show `verify` explicitly between `work` and `review`.
- Clarify that `spec` and `build` are alias entrypoints, while `plan` and `review` are direct lifecycle commands unless alias files are intentionally added later.

### 8. Verify That Spec Creation Support Is Already Present
Do not add new spec-generation surfaces unless the audit proves one is missing.
Instead, confirm and document the existing path:
- `/ai-eng/specify`
- `/spec`
- `skills/spec-driven-development/SKILL.md`
- `skills/prompt-refinement/SKILL.md`

Optional follow-up only if needed:
- decide whether to implement or remove the advertised `--template` option in `content/commands/specify.md`

## Risks
- Removing duplicate scripts without reference checks could break packaging or release flows.
- Removing anything from `docs-site/` too early could break the published docs surface.
- Fixing ADR paths in only one layer would increase drift across generated mirrors.
- Promoting `verify` without updating docs and rebuilt outputs would leave the workflow story half-fixed.

- `docs/` is explicitly documented as canonical.
- Stale root scripts and stale root docs are removed or archived safely.

- Existing spec support remains intact and clearly documented.

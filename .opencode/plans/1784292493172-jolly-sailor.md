# Skill Library Audit & Pruning Plan

**Goal:** Review every skill in the repo against the "skills should earn their place" framework (Every.io article), then act on the findings. Bar: **KEEP** = private context / custom tool access / specific custom workflow; **RETEST** = compensates for a model weakness (shelf life); **RETIRE** = restates public knowledge, duplicates another skill, or is plain-prompting-equivalent.

**Scope (user-locked):**
- `skills/` (127 git-tracked, repo-owned) → **active phased retirement**
- `skills-gtm/` (205 vendored LeadMagic, opt-in + gitignored, never auto-loads) → **advisory report only, no deletions**
- Build a **golden-dataset + LLM-judge proof harness** for borderline/RETEST survivors
- Add **governance** to prevent re-bloat

---

## Findings (verified, grounded in repo evidence)

**The cost surface is narrow.** Only **69 `model-invoked` skills** in `skills/` auto-load (~100 routing tokens each ≈ ~6.9k tokens of startup context). The **58 `user-invoked`** + all **205 `skills-gtm`** cost ~0 by default (enforced via `disable-model-invocation: true` + `metadata.category`, maintained by `sync-skill-taxonomy`). Build copies (`dist/`, `.opencode/`, `.claude/skills/`, `packages/`, `plugins/`) are gitignored — not audited.

**Headline numbers:**
| Set | KEEP | RETEST | RETIRE | %RETIRE |
|---|---|---|---|---|
| `skills/` (127) | ~22 | ~22 | ~61 | ~58% |
| `skills-gtm/` (205) | ~35 | ~10 | ~160 | ~78% |

**Key integrity finding:** `skills/incentive-prompting/SKILL.md:14` now explicitly *forbids* personas/stakes/challenge language — the exact techniques `AGENTS.md` + `skills/AGENTS.md` still advertise as "+45-115% quality." Live self-contradiction to fix during retirement.

**Repo already runs this framework:** `skills/DELETED_SKILLS.md` documents Batch 1 (9 stubs deleted 2026-05-31) with explicit criteria. We extend that convention. 66% of `skills/` already have `evals/`.

## Execution-readiness refinements (discovered during pre-execution verification)

These update the batches below — the reference sweep + build inspection changed what's safe to delete in-place vs. what needs a deliberate repoint.

**Build is a safety net.** `build.ts:209 validateCommandSkillReferences` + `build.ts:939 validateCanonicalSkills` validate every command→skill reference at build time. `bun run build` (run by `validate`) will **fail loudly** if a deleted skill is still hard-loaded by a command. So deletions are safe to attempt; misses surface as build errors. (Skill→skill and doc references are NOT auto-validated — handled manually below.)

**Wired skills — DO NOT delete in-place (defer to a repoint pass).** The reference sweep proved these are hard-loaded as the procedure of a command/agent. Per the article's own framework, "a skill that's the documented procedure for a command" = **specific custom workflow (KEEP)** — so they earn their place even though their *content* is public. Defer them, or repoint the command/agent to inline the guidance in a separate careful step:
- `code-simplification` ← `/simplify` ("Load `skills/code-simplification/SKILL.md`") + `ai-eng/simplify`
- `incremental-implementation` ← `/work` ("Load `skills/incremental-implementation/SKILL.md`")
- `spec-driven-development` ← `/spec` ("Load `skills/spec-driven-development/SKILL.md`")
- `thermo-nuclear-{code-quality,security,architecture,performance}-review` ← `/deep-review` dispatch table + the 4 reviewer agents
- `security-and-hardening` ← `security-scanner` agent ("Apply the `security-and-hardening` skill")
- `research-deep` ← **deployed** `agents/research-runner/` cron service (`ofelia.ini` → `run-research-deep.sh`) — KEEP outright
- `knowledge-capture` ← broad doc + `teach` refs — defer

**Safe-to-delete-now set (verified unwired, ~20 skills).** Only doc/index/tag/soft-mention refs to clean up:
- Stubs (7): `check-compiler-errors`, `fix-merge-conflicts`, `get-pr-comments`, `new-branch-and-pr`, `run-smoke-tests`, `what-did-i-get-done`, `review-and-ship`
- Pure-index domain, no command/agent load (11): `git-workflow-and-versioning`, `deprecation-and-migration`, `documentation-and-adrs`, `frontend-ui-engineering`, `api-and-interface-design`, `performance-optimization`, `context-budget`, `context-engineering`, `graphify`, `dependency-graph-analysis`, `idea-refine`
- Tag/soft only (2): `content-optimization` (only `tags:` refs), `verify-this` (soft mention in `control-ui`; survivor = `verification-loop`)
- Known skill→skill fixups on delete: update `monorepo-initialization` (drops `context-engineering`), update `control-ui` (drops `verify-this` mention).

**pstack cluster** (`principle-*` ×11 + root `principle-*` ×5 + `how`/`why`/`arena`/`reflect`/`automate-me`/`poteto-mode`): cohesive imported "style" set. `docs/attribution/cursor-plugins.md` already lists nearly all as **"Skipped (duplicate)"**. `poteto-mode` aggregates the principles by name, so the principle set must be deleted *with* a poteto-mode update (or together). `how`/`why`/`reflect` show massive false-positive prose matches (common English words) — treat distinctive skill refs only. Decision: retire the **16 principle skills** (11 pstack + 5 root) as one sub-batch after updating `poteto-mode`; leave `how`/`why`/`arena`/`reflect`/`automate-me`/`poteto-mode` itself for a separate decision (no clear unwired RETIRE signal beyond the attribution note).

**skills-gtm advisory — FINALIZED (from completed deep-read):** **33 KEEP · 16 RETEST · 156 RETIRE** (=205). KEEP core = leadmagic/(6) + tools/(15) + automation/(5: attio-setup, hubspot-setup, salesforce-setup, clay-automation, mcp-setup) + outbound/(4: email-deliverability, domain-infrastructure, inbox-setup, sending-platforms) + customer-success/(2: headless-support, support-tool-stack) + inbound/(1: website-visitor-identification). Demotions vs. earlier estimate: `crm-integration` + `api-enrichment` → RETEST (generic process layer). Biggest RETIRE block: `founder-led/` (41, 100% public frameworks). Advisory = report + keeplist only; **no deletions** (vendored; re-sync overwrites).

### Phase 2 — Phased retirement of `skills/` (build, 4 batches, each independently verifiable + reversible)

2. Append to `skills/DELETED_SKILLS.md` (Batch 2…) with rationale + migration path (extend Batch 1 format).
3. Repoint any references found in Phase 1's sweep (commands/agents/AGENTS.md).
4. `bun run build && bun run validate`.
5. Run `sync-skill-taxonomy` invariants check (every remaining skill has `metadata.category`; every user-invoked has `disable-model-invocation`).
6. Update `docs/reference/catalog-model.md` + `README.md` skill counts.

- Wire a `minimum-viable-prompting`-style "dead-weight" check into `sync-skill-taxonomy` (flag skills whose removal wouldn't change a sample task's output).
